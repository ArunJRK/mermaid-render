# Learnings

## Browser recovery probes

- Do not simulate WebGPU recovery by monkeypatching `Application.prototype.init` and attaching a fake `renderer.gpu.device.lost` promise. In this repo, that approach produced nondeterministic browser hangs and Playwright timeouts instead of a reliable recovery signal.
- Extract complex runtime-recovery branches into callable renderer methods first, then test those branches directly. For the WebGPU path, `_handleGpuDeviceLost()` is the right seam; forcing Pixi init state is not.
- Do not `await` a full renderer recovery path directly inside `page.evaluate()` when that path tears down and remounts GPU state. In this repo, `await window.__MERMAID_DEV__.runWebGpuRecoveryProbe()` still timed out even after switching away from the fake device-lost promise.
- Prefer a two-step browser probe for renderer recovery:
  1. trigger the recovery path without waiting on the full internal promise chain
  2. poll externally for observable completion signals such as warning emission, backend reattachment, and restored node count
- Keep the probe assertions on observable behavior, not on forced backend identity. In CI/browser automation here, proving that the WebGPU recovery branch re-renders successfully is more stable than proving the first backend was genuinely WebGPU.
- Headless Chromium in this environment exposes the WebGPU API surface but reports `No available adapters.` Treat that as an environment constraint, fail fast with an explicit status, and do not keep waiting for a WebGPU-only recovery path that cannot run there.

## Working pattern for this repo

- When a runtime path depends on Pixi/browser internals, add a deterministic harness in `packages/core/dev` and let Playwright assert on its outputs.
- When a probe fails by timeout, record the failed technique before trying the next one so resumed sessions do not repeat the same experiment.
- For non-orthogonal edges, fixed vertical label offsets are not robust. In this renderer, `midpoint.y - 10` was enough to make simple-flow labels intersect diagonal edges. Label placement needs to follow the local path tangent and offset along the preferred normal.
- Selection state should not survive graph rebuilds by accident. In this renderer, keeping `_selectedNodeId` across fold/focus/theme relayouts risks orphaned highlights because the old sprite tree is destroyed. Clear selection on rebuild unless there is an explicit remap policy.
- Hover-preview persistence should not depend only on Pixi container `pointerover`/`pointerout`. In this renderer, popup enter/leave was more reliable when tied to canvas-level pointer position against popup bounds, with hide scheduled on leaving the preview region.
- For deliberate overlap probes, clear any inherited hover state before moving into the overlap point. Repositioning sprites after render can otherwise leave stale hover on the previously hovered node and make the probe look nondeterministic.
- Bitmap font installation defaults are not a substitute for theme-driven text color. In this renderer, labels only started following philosophy switches once each `BitmapText` instance explicitly set `style.fill` from the active theme instead of inheriting the static atlas fill.
- Theme token cleanup is not enough by itself. After moving colors into `theme.ts`, add browser assertions against live Pixi state to prove philosophy switches actually recolor existing node, edge, subgraph, and broken-link visuals instead of only affecting newly created objects.
- Do not assume nested Mermaid subgraphs survive into the render model with a reliable explicit parent-child hierarchy. In this repo, the safer proof for theme depth fills was a dedicated `SubgraphContainer` browser probe rather than a browser assertion that depended on parsed nested-subgraph ancestry.
- Contrast misses in hand-tuned dark palettes tend to hide at the margins, especially on deeper subgraph tints. In this repo, the only reliable way to catch them was a direct contrast regression over the actual theme colors, not visual intuition.
- Pixi background color can come back as a color object in browser probes, not a raw number. Normalize debug output before asserting exact palette values.
- Once the renderer starts honoring `prefers-color-scheme` by default, browser screenshot baselines need an explicit test color scheme. Otherwise the same test can flip palettes under a different host/browser default.
- When a status is supposed to be non-color-only, assert the glyph or shape difference directly in debug state and then update the screenshot baseline intentionally. Otherwise a visual-only tweak can look correct by eye but still remain unverified.
- In a large serial Playwright file, default per-test timeouts become the bottleneck before the renderer does. Raise the file-level timeout rather than weakening invariants when the failures are clearly harness-side.
- `@mermaid-js/parser` is not a drop-in replacement for this repo's supported Mermaid subset. Its public parse surface does not cover `flowchart`, `classDiagram`, or `stateDiagram`, so switching to it would reduce bundle size at the cost of breaking v1 scope.
- `import.meta.glob(..., { eager: true, query: '?raw' })` for the demo examples bakes the entire `.mmd` corpus into the initial browser entry chunk. Keep example files lazy and resolve them through loader functions plus a small cache instead.
- Do not keep a bundle split just because `index-*.js` gets smaller. Check the full initial preload path in `dist-demo/index.html`; a tiny entry chunk plus a preloaded multi-hundred-kilobyte core chunk is not an actual improvement.
- When a renderer already has baseline dimming rules, relationship emphasis must override that baseline for the active set. In this repo, reusing `Math.min(...)` with Narrative's spine dimming left the hovered node stuck at `0.7` alpha until the active related set was treated as a full-opacity exception.
- For interaction-state regressions, expose live `alpha` values in the browser debug API instead of relying only on screenshots. The connected-neighbor emphasis check was much easier to stabilize once node and edge alpha were directly readable in Playwright.
- Do not run artifact production and artifact verification in parallel when the verifier reads the build output directory. In this repo, running `build:demo` and `bundle:check` concurrently reported the previous demo chunk instead of the freshly built one.
- Paint-order claims need both code alignment and browser-visible indices. In this repo, item 28 was not actually satisfied until the node internals were reordered so the badge sat below hover/selection overlays, and the dev API exposed child/stage indices for direct assertions.
- Watch for uncancelled local `requestAnimationFrame` loops even when a bigger animation subsystem exists. In this repo, the live relayout path was not using `LayoutAnimator`, but it still had an interrupt bug because `_relayout()` owned its own fade loop with no generation token.
- Degenerate-input proof needs an inline-source harness, not only bundled fixture files. In this repo, item 32 only became testable once the dev API could load raw Mermaid source strings directly, because empty graphs and deliberately pathological labels do not belong in the normal example corpus.
- If a visual invariant is important enough to discuss as a screenshot-worthy issue, add a persistent Playwright snapshot baseline for it instead of relying only on transient local screenshots. In this repo, the degenerate inline-label case now has a committed baseline alongside the broken-link badge case.
- Resize-observer driven refits need explicit size-delta gating. In this repo, wiring `fitToView()` directly to every observer callback caused flaky interaction tests until the callback ignored unchanged width/height pairs.
- Browser hover helpers should retry against current rendered bounds after layout-driven changes. In this repo, the hover-glow test stopped flaking only after `hoverNode()` re-read node bounds and verified `hoverVisible` instead of assuming one mouse move would land on the active sprite.
- Mermaid nested subgraphs cannot be treated as "nodes only" when computing container bounds. In this repo, parent containers could miss child container extents even though the leaf nodes fit, so the fix had to union already-computed child subgraph bounds into the parent bounds pass.
- For nested-container assertions, use the immediate rendered parent rather than every transitive ancestor. In this repo, the real clipping contract was parent-to-child containment; checking all ancestor pairs added noise without proving the right thing.
- For spring safety, `Number.isFinite(...)` is not enough by itself. In this repo, the integrator could explode to absurd but still finite magnitudes, so the guard had to treat unreasonable state magnitude as a snap-to-target condition too.
- Bounded animation time belongs in the spring primitive, not only in higher-level loops. Once `maxDuration` lived in `spring.ts`, both unit-level springs and viewport-driven browser transitions inherited the same no-run-forever contract.
- Cancelling an animation is not cleanup unless it resolves pending visual side effects. In this repo, `LayoutAnimator.cancel()` had to actively remove fade-out sprites and normalize fade-ins, otherwise interrupting one animation with the next still left orphaned or partial-alpha objects on stage.
- For interruption bugs, add a focused unit seam even if the higher-level runtime path is only partially active. Here, `layout-animator.test.ts` was the right place to pin the explicit `cancel()` contract while the broader relayout path remains split across other code.
- Release verification should prove the publish surface, not just the build graph. In this repo, adding `npm pack --dry-run` to the normal `verify:core` gate closed a real blind spot in item 12: CI now checks the package payload, not only TypeScript/tests/build success.
- For preview-cache and delayed-hover bugs, expose tiny deterministic hooks in the dev harness instead of guessing with production example timing. In this repo, preview override source/delay hooks made it possible to prove both stale-result suppression and reload invalidation in Playwright.
- Empty-canvas interaction tests should choose a point proven to be outside rendered node bounds, not assume a fixed corner is always empty. In this repo, the deselect regression was test fragility caused by a hardcoded click point, not a renderer selection bug.
- Last-write-wins tokens only work if ownership is claimed before the first awaited step. In this repo, `loadFile()` incremented its generation token after `readExampleFile(...)`, which let a stale slower load reclaim the view after a newer load had already won.
- “Uses the target file’s own philosophy” includes font family, not only layout and colors. In this repo, cross-file previews still misrepresented blueprint targets until the mini preview labels switched from hardcoded `MermaidNode` to philosophy-aware font selection.
- Unsupported examples in the visible demo corpus are release bugs, not harmless fixtures. In this repo, leaving `examples/blueprint-classes.mmd` exposed made the browser overlap sweep look like a rendering failure when the actual issue was unsupported `classDiagram` syntax.
- Hiding an unsupported example from the UI is not the whole scope correction if the build still ships it as a lazy asset. In this repo, the demo only fully matched the v1 parser boundary once the import glob itself excluded `blueprint-classes.mmd`, not just the rendered file list.
- Philosophy docs can discuss future fit, but README/demo surfaces must match shipped parser reality. In this repo, the right correction was to narrow release claims to `flowchart` support and hide unsupported examples, not to pretend class-diagram support existed.
- When an issue is visual enough to justify “take a photo,” store it as a committed Playwright snapshot baseline. Local one-off screenshots are useful during debugging, but they do not protect future regressions unless they live in `tests/browser/render.spec.ts-snapshots/`.
- Router occupancy must use the same rendered-footprint rule as Pixi node drawing. In this repo, reserving only layout width let Blueprint wires pass through visibly expanded long-label nodes until the occupancy grid and wire registry both switched to the shared footprint helper.
- “No dropped wire” is a separate contract from “found an A* path.” In this repo, the correct fallback was to emit a visible direct route and mark the route result congested, not to pretend routing succeeded or silently omit the edge.
- Browser proofs for routing should assert the real invariant, not one particular path shape. In this repo, “wire does not intersect the expanded node footprint” was stable and correct; asserting a specific offset segment shape overfit the current router output.
- Dimmed-context quality needs its own contract instead of borrowing full text-contrast rules. In this repo, a realistic rule was: dimmed text stays readable as context, and dimmed alpha stays clearly above hidden (`alpha 0`), rather than pretending focus-dim should meet normal foreground-text contrast targets.
- For active animation-path cleanup, move the shipped runtime first and leave the dormant path second. In this repo, the meaningful improvement for item 50 was to put viewport motion and relayout fade on the Pixi ticker before worrying about the older `LayoutAnimator` branch that still is not the live relayout engine.
- When a runtime warning is part of the user contract, do not let the dev probe overwrite it with a later `info` status. In this repo, the Blueprint fallback warning was already correct; the harness was the part hiding it.
- If a relayout starts animating in place instead of rebuilding immediately, live theme changes have to move with it. In this repo, the first pass updated moving edge geometry but left node badges and subgraph labels on the old philosophy colors until the final rebuild, which broke the live recolor contract.
- Interrupted relayouts need to restore viewport alpha explicitly, not only cancel the old callback. In this repo, moving to ticker-driven motion exposed that a previously started fade could still leave `viewport.alpha = 0.3` behind unless the new path forced it back to `1`.
- For edge-attachment proofs during animation, use a rectangle-only browser probe and assert against moving node centers plus display width/height. That kept the test stable without overfitting to a specific curved-path shape.
- When an older helper remains in the tree but the shipped runtime has already moved to ticker-driven animation, remove the helper's private timing policy too. In this repo, adding `LayoutAnimator.attachTicker(...)` was a small but worthwhile step because it prevents a dormant rAF loop from silently reintroducing item-50 drift later.
- For trust-boundary requirements, browser proof should assert the negative behavior directly. In this repo, the meaningful runtime proof for item 36 was not just “the resolver returns null,” but “an out-of-scope URL target renders broken, warns clearly, and performs zero raw fetches.”
- For stale-sprite regressions, inspect the live Pixi stage directly instead of inferring cleanliness from map sizes alone. In this repo, a dedicated `getSceneInventory()` probe made it possible to prove there were no orphaned or duplicate node/edge/subgraph sprites after fold/focus/philosophy rebuilds.
- For transition-quality checks, sample the renderer in the middle of a known live animation, not only before/after. In this repo, item 30 needed both a settled-stage inventory check and a mid-animation inventory check to say anything meaningful about double-draw risk.
- For “no teleport” claims, compare one moving node across several sampled frames and assert monotonic progress from the start, not exact frame-by-frame coordinates. In this repo, that produced a stable continuity check without overfitting to one easing curve.
- For stress-mode simplification, expose the exact rendered detail flags in the browser debug API and assert them directly. In this repo, checking `labelVisible`, `chevronVisible`, and `badgeVisible` was much more reliable than trying to infer suppression from screenshots alone.
- When a philosophy only offers best-effort routing, make the failure mode explicit in the shipped runtime. In this repo, item 18 became much more honest once non-Blueprint edge/node crossings emitted a readable `EDGE_NODE_CROSSING` warning instead of staying a docs-only caveat.
- Font-switch completeness needs browser proof across every text surface, not just node labels. In this repo, Blueprint was only truly consistent once subgraph text, edge labels, and hover-preview title text were all checked against their live font families.
- For degradation policy, prove the behavioral suppression itself in-browser, not just the resulting warning state. In this repo, item 45 became stronger once stress mode was shown to suppress hover previews on a large linked graph, not merely emit `PERF_STRESS_THRESHOLD`.
- If a helper and the shipped runtime are supposed to share one animation model, remove the helper's last fallback clock too. In this repo, item 50 only really stopped drifting once both `Viewport` and `LayoutAnimator` switched to Pixi ticker semantics even outside mounted-app contexts, using `Ticker.shared` instead of private `requestAnimationFrame` branches.
- When a degradation mode changes visible density, keep a committed canvas snapshot as well as debug-flag assertions. In this repo, stress-mode suppression was better protected once the test kept both the `labelVisible`/`badgeVisible` checks and a baseline screenshot of the simplified canvas.
- For motion-quality claims, inventory checks are not enough on their own. In this repo, item 30 got meaningfully stronger only after adding a committed mid-relayout canvas snapshot, captured while nodes were measurably moving, so the suite checks the actual drawn frame instead of only object counts and coordinates.
- Snapshot tests around relayout/fold visuals need an explicit post-transition settle point, not just the logical state flip. In this repo, the fold-after-philosophy-switch snapshot was flaky until the test waited for `viewportAlpha === 1` and a short extra settle window before capturing the canvas.
- If a helper no longer participates in the shipped runtime, delete it instead of continuing to “align” it. In this repo, item 50 only became structurally honest once the dormant `LayoutAnimator` branch was removed rather than kept around as a second non-authoritative animation subsystem.
- Responsive shell layout should key off actual content height, not a guessed mobile offset. In this repo, the mobile toolbar could outgrow the hardcoded `230px` canvas offset, so the durable fix was to place the canvas below the real toolbar bottom and keep that synced as the toolbar content changed.
- For animated geometry assertions, the tolerance should match the rendered invariant, not one exact sampled interpolation point. In this repo, the moving-edge attachment check was more stable once the boundary tolerance allowed small intermediate-frame drift while still proving the endpoint stayed attached to the moving node boundary.
- After stabilizing a known flaky browser assertion, rerun the entire release gate instead of stopping at focused repeats. In this repo, the honest confirmation point was not the repeated relayout test alone, but a fresh green `pnpm verify:core` run with all `53` browser tests passing on the same tree.
- If a warning is part of the author contract, prove it through the shipped browser harness instead of stopping at parser/unit coverage. In this repo, malformed `%% @link ...` syntax only really closed item 34 once Playwright asserted the status UI surfaced `LINK_DIRECTIVE_INVALID` as readable feedback.
- A cache-size constant is not enough evidence for an LRU claim. In this repo, item 38 only became well-supported once cache hits refreshed recency and a browser probe proved the `13th` preview evicted the untouched oldest target while retaining a recently touched one.
- Resolver-contract items need browser proof through the real navigation path, not just path-helper unit tests. In this repo, item 35 became much stronger once Playwright proved relative `../` plus `.` segments, absolute paths, and extensionless spellings all landed on the same canonical `.mmd` target and revealed the same node.
- When a feature has both a helper path and a shipped interaction path, prove the shipped interaction path separately. In this repo, item 33 was not fully convincing until Playwright exercised an actual linked-node click and showed the fragment target node was selected after `link:navigate`, not just after a direct helper call.
- Lifecycle contracts should prove the allowed no-op path as well as the error paths. In this repo, item 41 was cleaner once Playwright asserted that remounting the same live renderer onto the same canvas succeeds harmlessly, while remounting to a different canvas and stealing an owned canvas still fail clearly.
- “WebGPU preferred” is not fully proven by backend logging alone. In this repo, item 43 needed an explicit browser probe where `navigator.gpu` existed but `requestAdapter()` returned `null`, and the renderer still mounted and rendered through WebGL instead of failing.
- Broken-link coverage should distinguish missing-file and missing-fragment failures, not treat them as interchangeable. In this repo, item 34 got stronger once Playwright proved an existing file with a missing target node produced a broken badge, a `false` click result, and a readable `LINK_TARGET_NODE_NOT_FOUND` warning.
- Generic init-failure coverage is not the same as a no-backend contract. In this repo, item 42 became stronger only after Playwright exercised a probe that failed with `No supported GPU backend` specifically and still painted the readable fallback canvas state.
- Recovery-contract probes need to use a real node id from the fixture and measure the post-recovery scene from outside the helper. In this repo, the first item-27 attempt looked broken because the probe targeted `OrderSvc`, which is not a node in `order-service.mmd`; using `orderFlow` and checking bounds after the recovery call produced the real signal.
- When a zoom-range contract exists in code, prove the exact clamp values directly instead of inferring them from softer layout behavior. In this repo, item 26 became clearer once Playwright asked for absurdly high and low relative zoom values and asserted the renderer stopped at `5.0` and `0.1`.
- When long-label handling is "grow the node," assert both containment and visible width growth. In this repo, item 14 became much clearer once Playwright proved the long rectangular node contained the full label bounds, widened materially versus a short peer, and kept a snapshot baseline for the expanded shape.
- For low-zoom readability claims, one representative diagram is not enough. In this repo, item 16 only became convincing once Playwright drove the full shipped example corpus and the stress graph down to the minimum zoom clamp and checked rendered label bounds directly there.
- For edge-ending claims, route-point checks are not enough on their own. In this repo, item 17 got stronger once `EdgeGraphic` exposed the drawn arrow tip and angle, so Playwright could prove the visible arrowhead sits on the boundary endpoint and follows the final segment direction.
- For non-color state cues, verify the mixed-state case instead of only the base state. In this repo, item 57 got stronger once Playwright proved the broken badge stayed distinguishable while that same node was also selected and hovered, with a committed snapshot rather than only a glyph enum check.
- For emphasis-state distinctions, keep visual baselines for both the isolated and coexistence cases. In this repo, item 60 became much harder to regress once Playwright kept clipped snapshots for `selection only` and `selection + hover`, instead of relying only on `selectionAlpha` and `hoverAlpha`.
- For relationship-emphasis claims, keep one visual artifact for hover and one for selection. In this repo, item 61 became more convincing once Playwright kept separate full-canvas snapshots for hover-driven and selection-driven neighbor emphasis, instead of relying only on per-node and per-edge alpha assertions.
- For perceptibility floors, prove the weakest theme in a real browser as well as in unit math. In this repo, item 62 became more trustworthy once Playwright captured clipped `breath` snapshots for hover-only and selection-only states, rather than trusting the low-alpha contrast calculation alone.
- For system-theme behavior, keep before/after rendered baselines as well as palette-number assertions. In this repo, item 56 became stronger once Playwright kept a system-light narrative snapshot and a post-`setThemeMode('dark')` snapshot, rather than only checking background and fill color values.
- For dimmed-context claims, prove a real scene band in-browser instead of only a theme blend calculation. In this repo, item 63 became more defensible once Playwright asserted the unrelated node stayed between `0.3` and `0.5` alpha in light narrative mode and kept a full-canvas baseline for that selection-driven context view.
- For live recolor claims, keep snapshots for both the main graph and a status-bearing special case. In this repo, item 31 got stronger once Playwright kept a narrative-before / blueprint-after pair on the overview graph and a separate radial broken-link baseline, instead of relying only on numeric fill/font changes.
- For depth-tint claims, pair the cross-theme fill-value probe with one real nested-scene baseline. In this repo, item 59 became more persuasive once Playwright kept a nested `map` subgraph screenshot alongside the browser metrics that already proved three distinct depth fills per philosophy.
- For emphasis z-order claims, use the deliberate overlap probe and keep artifacts for both hover and selection. In this repo, item 64 became much harder to regress once Playwright kept clipped overlap snapshots showing the top node still owns the visible hover or selection treatment over the occluding sibling.
- Paint-order claims need both local and stage-level proof. In this repo, item 28 was still incomplete until the browser metrics exposed subgraph and edge layer indices, so Playwright could assert `subgraphs < edges < nodes` instead of only checking node internals and the preview popup.
- For visually distinctive edge-shape claims, keep a committed snapshot as well as geometry assertions. In this repo, item 20 became more defensible once the self-loop and opposite-direction edge fixture had a baseline screenshot, not just width/offset checks on the computed bounds.
- For label-clearance claims, a blueprint screenshot can catch visual regressions that still satisfy coarse geometry rules. In this repo, item 19 became better protected once the Blueprint `simple-flow` canvas had a committed snapshot baseline alongside the label-vs-node and label-vs-path overlap assertions.
- For order-independence claims, do not key browser proof off parser-generated edge ids when source order changes those ids. In this repo, item 54 became stable once Playwright normalized Blueprint routes by `source->target` and compared rounded rendered segments instead.
- For delayed-overlay positioning, change the view during the debounce window and assert the final anchor, not just visibility. In this repo, item 37 was only properly covered once Playwright zoomed mid-hover and proved the preview used the node's current on-screen position instead of stale pre-zoom coordinates.
- For preview-flicker claims, move the pointer across the actual node-to-popup gap in small timed steps instead of only teleporting into the popup. In this repo, item 23 was only directly covered once Playwright walked that path and proved the preview stayed visible throughout the bridge, not just at the endpoints.
- For stale-anchor claims, cover viewport translation separately from zoom. In this repo, item 37 needed both checks: one where scale changed and one where only viewport position changed, because a preview path that survives zoom can still accidentally reuse stale pre-pan screen coordinates.
- For deployable-demo claims, smoke-test the built artifact from a plain static server, not only the dev server. In this repo, item 3 became more honest once Playwright hit `dist-demo` through a tiny static host and proved the built app still rendered, switched philosophy, followed a cross-file link, and responded to the file picker.
- For browser proofs about coexistence state on linked or broken-link nodes, separate “selection exists” from “click navigation works.” In this repo, suite stability improved once the state-only proofs used `window.__MERMAID_DEV__.selectNode(...)` instead of relying on click selection for nodes that also carry link behavior.
- For low-glow hover perceptibility checks, use a plain non-linked node if the requirement is about theme visibility rather than link affordances. In this repo, `Gateway` was a better `breath` hover probe than linked `OrderSvc`, which carried extra badge/link behavior unrelated to item 62.
- After a browser-suite stabilization pass, rerun the full repo gate rather than only the browser slice. In this repo, the meaningful confirmation point was a fresh green `pnpm verify:core`, because it rechecked lint, typecheck, unit, browser, build, built-demo smoke, bundle budgets, and pack output on the same tree.
- If a requirement cluster matters on its own, give it a first-class browser command instead of forcing manual `playwright -g ...` spelunking. In this repo, the runtime/GPU recovery surface became easier to audit once the lifecycle slice was exposed as `pnpm --filter @mermaid-render/core test:browser:lifecycle`.
- If a screenshot is intended for the README, pair it with a browser snapshot test and then copy the generated artifact into a docs asset path. In this repo, that kept the public README images tied to reproducible Playwright output instead of ad hoc manual captures.
- For non-rectangular shape-fit claims, keep one committed canvas baseline as well as the inscribed-area geometry assertion. In this repo, item 15 became harder to regress once the `shape-showcase` fixture produced a Playwright snapshot in addition to the `circle`/`diamond`/`hexagon` containment math.
- For failure-state requirements, keep a snapshot of the degraded-but-readable canvas, not only status flags. In this repo, the invalid-Mermaid path became better protected once the suite kept an `invalid-mermaid-error-state` baseline in addition to asserting `statusLevel = error` and overlay text.
- For backend-unavailable and init-failure states, keep separate baselines for each user-visible fallback path. In this repo, item 42 was better covered once Playwright kept one canvas baseline for generic renderer-init failure and another for the explicit no-GPU-backend case, instead of treating all fallback rendering as one generic state.
- Broken-link coverage should keep distinct artifacts for missing-file and missing-fragment states when the rendered badge path differs only by runtime cause. In this repo, item 34 became stronger once the existing missing-file badge snapshot was joined by a dedicated missing-fragment clipped snapshot.
- Author-warning paths benefit from full-canvas baselines, not only text assertions. In this repo, the malformed-`@link` case became better protected once Playwright kept a `malformed-link-warning-state` canvas snapshot alongside the status-message check.
- Trust-boundary warnings deserve the same treatment as other visible failure states. In this repo, item 36 became better protected once the out-of-scope link rejection kept a full-canvas warning baseline, not only a broken-badge assertion and a `fetchCount === 0` check.
- For successful fallback paths, add a visual-probe seam instead of trying to snapshot after the normal probe already destroyed the renderer. In this repo, item 43 needed a dedicated lifecycle-harness visual probe so Playwright could capture the WebGL-rendered fallback canvas after `requestAdapter()` returned `null`.
- Source-level guardrails should match the real directory shape, not only the current file layout. In this repo, item 58 was safer once the hardcoded-color regression test recursed through `src/renderer/**` instead of assuming all renderer sources lived at one flat level.
- Guarantee-level scope calls belong in the public README as well as technical docs. In this repo, item 51 was still too implicit until the README stated that only `blueprint` is collision-aware and the other shipped philosophies should not be read as obstacle-free routers.
